## Краткое описание работы
В ходе лабораторной работы изучены принципы архитектуры REST, создан полноценный Controller-based API с поддержкой CRUD-операций. Освоены различия между Minimal API и Controller-based API, работа со Swagger UI для автоматической документации и тестирования, а также использование расширения REST Client в VS Code. Реализованы DTO для защиты API от некорректных данных, настроены правильные HTTP-статусы ответов и добавлены дополнительные маршруты (поиск, фильтрация по приоритету, статистика, сортировка). Настроен CORS для будущего взаимодействия с фронтендом.

## Список реализованных маршрутов

| HTTP метод | URL | Действие |
|------------|-----|----------|
| GET | /api/tasks | Получить все задачи (поддерживает фильтрацию по параметру `completed`) |
| GET | /api/tasks/{id} | Получить задачу по идентификатору |
| POST | /api/tasks | Создать новую задачу (принимает `CreateTaskDto`) |
| PUT | /api/tasks/{id} | Полностью обновить существующую задачу (принимает `UpdateTaskDto`) |
| DELETE | /api/tasks/{id} | Удалить задачу |
| PATCH | /api/tasks/{id}/complete | Переключить статус выполнения задачи (инвертирует `IsCompleted`) |
| GET | /api/tasks/search?query={text} | Поиск задач по подстроке в заголовке или описании (без учёта регистра) |
| GET | /api/tasks/priority/{level} | Фильтрация задач по приоритету (`Low`, `Normal`, `High`) |
| GET | /api/tasks/stats | Получить статистику: общее количество задач, выполненных и невыполненных |
| GET | /api/tasks/sorted?by=priority&desc={true/false} | Сортировка задач по приоритету (возможен порядок по возрастанию/убыванию) |

## Итоговая таблица ASP.NET Core Controller-based API

| Аспект | ASP.NET Core Controllers |
|--------|--------------------------|
| Маршруты | `[HttpGet]` атрибут над методом |
| Группировка маршрутов | Класс-контроллер |
| Базовый URL | `[Route("api/[controller]")]` |
| Параметр пути | `(int id)` — параметр метода |
| Параметр запроса | `[FromQuery] bool? completed` |
| Тело запроса | `[FromBody] CreateTaskDto dto` |
| Ответ 200 | `return Ok(data)` |
| Ответ 201 | `return CreatedAtAction(...)` |
| Ответ 404 | `return NotFound(...)` |
| Ответ 204 | `return NoContent()` |
| Типизация данных | Строгая (C#) |
| Документация | Swagger — устанавливается отдельно (`Swashbuckle.AspNetCore`) |

## Главные выводы

1. **REST — это архитектурный стиль, а не протокол.** Принципы (stateless, единый интерфейс, кэшируемость) универсальны и применимы в любом языке и фреймворке.

2. **Контроллер в ASP.NET Core выполняет роль роутера (как в Express), но с автоматической документацией и строгой типизацией.** Использование атрибутов `[HttpGet]`, `[Route]` и встроенной привязки параметров упрощает разработку и снижает количество ошибок.

3. **DTO защищают API от некорректных данных.** Разделение моделей на входные (CreateTaskDto, UpdateTaskDto) и внутренние (TaskItem) позволяет серверу контролировать, какие поля клиент может изменять, а какие генерируются автоматически (Id, CreatedAt).

4. **Swagger UI — мощный инструмент для тестирования и документации.** Он позволяет выполнять запросы к API прямо из браузера без использования сторонних программ (Postman) и без написания тестового JavaScript-кода.

5. **Правильные HTTP-статусы являются частью контракта API.** Возврат 201 при создании, 404 при отсутствии ресурса, 204 при успешном удалении — клиент (например, JavaScript `fetch`) по коду ответа сразу понимает результат операции.
